## 3. Scoring Methodology

### Factuality Scoring

Scored by the `score_factuality` function in `llm_client.py`. This function sends a meta-evaluation prompt to Claude Sonnet 4.6:

```
You are evaluating AI output quality. Compare the expected output with the actual output and rate their factual similarity on a scale of 0-100.

Expected output:
{expected}

Actual output:
{actual}

Respond with ONLY a single integer from 0 to 100. No other text.
```

- max_tokens: 10
- The response is parsed via regex (`\d+`) to extract the first integer found.
- The score is clamped to 0-100 with `min(max(score, 0), 100)`.
- On any exception, defaults to 0.0.
- This is an LLM-as-judge pattern where Claude evaluates its own output quality.

### Format (JSON) Scoring

Binary scoring. The LLM response is passed through `json.loads()`:
- If parsing succeeds: score = 100.0
- If `JSONDecodeError` or `TypeError`: score = 0.0

### Aggregate Scores

- **Quality Score** = mean of all individual test scores for a run (0-100)
- **Factuality Score** = mean of factuality test scores only (0-100, or null if none)
- **Format Score** = mean of format_json test scores only (0-100, or null if none)
- **Drift Flag** = true if quality score < 75% (configurable via `DRIFT_THRESHOLD` environment variable)

## 4. Per-Test Tracking

Every evaluation run stores granular per-test-case results in the `eval_results` table using the `EvalResult` model.

### EvalResult Schema

| Column | Type | Description |
|--------|------|-------------|
| id | Integer (PK) | Unique result identifier |
| eval_run_id | Integer (FK) | Links to the parent EvalRun |
| test_case_id | Integer (FK) | Links to the specific EvalTestCase |
| response_text | Text | Raw LLM response for this test case |
| score | Float | Score for this individual test (0-100) |
| latency_ms | Float | Time taken for the LLM call in milliseconds |
| status | String | "success" or "error" (set to "error" if response starts with "ERROR:") |
| created_at | DateTime | Timestamp of result creation |

### Relationships

- Each `EvalRun` has many `EvalResult` entries (one per test case in the run).
- Each `EvalResult` links back to its `EvalTestCase` for category and prompt information.
- The drift-check endpoint aggregates historical `EvalResult` scores per test case across runs to compute per-test trends.

## 5. Drift Detection Algorithm

### Threshold

The drift threshold is 75% by default, configurable via the `DRIFT_THRESHOLD` environment variable. A quality score below this threshold triggers the drift flag.

### Severity Levels

| Severity | Criteria | Recommended Action |
|----------|----------|-------------------|
| **none** | Score >= threshold + 10 (i.e., >= 85%) AND trend is not declining | No action needed |
| **warning** | Score within 10 points of threshold (75-85%) OR trend is declining | Investigate; re-run evaluation to confirm |
| **critical** | Score < 75% threshold OR sudden drop > 15 points from previous average | Create incident with severity high; run troubleshooting checklist |

### Sudden Drop Detection

If at least 3 historical runs exist, the system computes the average of all previous scores (excluding the current run). If the current score is more than 15 points below this average, severity is escalated to "critical" regardless of the threshold.

### Trend Analysis (`_compute_trend`)

1. Collect quality scores from the most recent N runs (configurable window, default 5, range 2-20).
2. Sort scores in chronological order (oldest first).
3. Split scores into two halves at the midpoint.
4. Compute the average of the first half and the second half.
5. Calculate the difference: `second_half_avg - first_half_avg`.
6. Classify:
   - Difference > 3.0 points: **improving**
   - Difference < -3.0 points: **declining**
   - Otherwise: **stable**

During evaluation runs, if the trend is "declining" AND the current score is within 10 points of the drift threshold, the drift flag is set to true even if the score has not yet crossed the threshold.

In the drift-check endpoint, trend is computed for both the overall service and for each individual test case.

### Variance Calculation (`_compute_variance`)

1. Collect quality scores from the most recent N runs (same window as trend).
2. Compute the mean of all scores.
3. Calculate population variance: `sum((score - mean)^2 for each score) / count`.
4. Round to 2 decimal places.

Interpretation:
- Low variance (< 25): scores are consistent; model behavior is predictable.
- Medium variance (25-100): some fluctuation; may indicate intermittent issues.
- High variance (> 100): significant instability; investigate test reliability and model behavior.

### Confidence Scoring

| Run Count | Confidence | Meaning |
|-----------|------------|---------|
| 1-2 runs | low | Insufficient data for reliable trend analysis |
| 3-4 runs | medium | Trend direction is indicative but may not be reliable |
| 5+ runs | high | Sufficient data for reliable trend and variance analysis |

## 6. Dashboard Metrics

### Latency Percentiles

| Metric | Source | Description |
|--------|--------|-------------|
| P50 | ConnectionLog / APIUsageLog | Median latency (50th percentile) |
| P95 | ConnectionLog / APIUsageLog | 95th percentile; captures tail latency |
| P99 | ConnectionLog / APIUsageLog | 99th percentile; captures worst-case scenarios |

Percentiles are computed by sorting all latency values and selecting the value at the corresponding index position.

### Error Categorization

All API errors are classified and logged in `api_usage_log` with the following statuses:
- `error_timeout` -- connection timeout
- `error_rate_limit` -- Anthropic rate limit hit
- `error_server` -- Anthropic 5xx response
- `error_auth` -- authentication failure
- `error_bad_request` -- malformed request
- `error_unknown` -- uncategorized exception
- `blocked_safety` -- blocked by safety scanner
- `retry_0`, `retry_1` -- retry attempt indicators

## 7. Known Limitations

- Small dataset (2 test cases per service) -- sufficient for demonstration but not production-grade.
- Factuality scoring depends on Claude's own judgment (LLM-as-judge), which may have blind spots for its own outputs.
- JSON format checks use simple `json.loads()` -- they do not validate schemas or expected key values.
- All test cases are in English only.
- Test cases do not cover edge cases such as very long responses, refusals, or adversarial inputs.
- Variance and trend calculations require multiple runs to be meaningful (confidence is "low" with fewer than 3 runs).
- A healthy AI service should score 90%+ consistently. Scores between 75-90% warrant investigation. Scores below 75% trigger critical drift severity and should result in an incident.
